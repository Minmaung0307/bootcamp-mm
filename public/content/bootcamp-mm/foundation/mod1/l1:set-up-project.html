Set Up the Project

Introduction

Full-stack web developer တစ်ယောက်ဖြစ်ဖို့ ခရီးစတင်လိုက်တဲ့အတွက် ကြိုဆိုပါတယ်! အခုလာမယ့် တစ်ပတ်အတွင်းမှာ developer တစ်ယောက်ဖြစ်ဖို့ အခြေခံအကျဆုံးနဲ့ အရေးကြီးဆုံးဖြစ်တဲ့ HTML, CSS နဲ့ Git နည်းပညာ သုံးခုကို ကျွမ်းကျင်အောင် သင်ယူရမှာပါ။

ဒီပညာရပ်တွေကို Run Buddy ဆိုတဲ့ ကုမ္ပဏီအတွက် landing page တစ်ခု တည်ဆောက်ရင်းနဲ့ လက်တွေ့လေ့လာသွားမှာပါ။ မင်းကို freelance web developer တစ်ယောက်အနေနဲ့ ငှားထားတယ်လို့ သဘောထားပြီး ဒီ website ကို အလှပဆုံးဖြစ်အောင် ဖန်တီးပေးရမှာ ဖြစ်ပါတယ်။ အပြင်အဆင်ပိုင်း (design) ပြီးသွားရင်တော့ ကုမ္ပဏီရဲ့ developer တွေဆီကို လက်လွှဲပေးရမှာပါ။

ဒီသင်ခန်းစာမှာ ဘာတွေတတ်မြောက်မလဲ?

ပထမဆုံး သင်ခန်းစာမှာတင် HTML code တွေ စရေးနိုင်ဖို့အတွက် အရေးကြီးတဲ့ tools တွေနဲ့ လုပ်ငန်းစဉ်တွေကို လေ့လာရမှာပါ။ အဓိကအားဖြင့် အောက်ပါတို့ကို လုပ်ဆောင်နိုင်မှာ ဖြစ်ပါတယ် -

  • Command line ကို သုံးပြီး folder တွေထဲ သွားလာတာနဲ့ file အသစ်တွေ တည်ဆောက်တာ။
  • HTML document တစ်ခုရဲ့ အခြေခံ structure ကို စနစ်တကျ ချမှတ်တာ။
  • Git ကို သုံးပြီး ကိုယ့်ရဲ့ အလုပ်တွေကို သိမ်းဆည်းတာနဲ့ remote repository ပေါ်ကို push လုပ်တာ။

Preview (အကြိုကြည့်ရှုခြင်း)

ဒီအပတ်မှာ ဘာတွေ တည်ဆောက်မလဲဆိုတာ အကြမ်းဖျင်း ကြည့်လိုက်ရအောင်။ Run Buddy ဒီဇိုင်းအဖွဲ့ကနေ Landing page အတွက် Mock-up (ပုံစံကြမ်း) တစ်ခု ပေးထားပါတယ်။

အရေးကြီးချက်: Run Buddy စာမျက်နှာကို လုပ်နေတဲ့အချိန်မှာ ဒီပုံကို ခဏခဏ ပြန်ကြည့်ရမှာဖြစ်လို့ သင့်ကွန်ပျူတာထဲမှာ Download ဆွဲထားဖို့ အကြံပြုပါတယ်။ ပုံပေါ်မှာ ညာဘက်ထောက် (Right-click) နှိပ်ပြီး "Save Image As" နဲ့ မှတ်မိလွယ်မယ့်နေရာမှာ သိမ်းထားလိုက်ပါ။

သင်ဟာ Developer အသစ်တစ်ယောက်အနေနဲ့ ဒီ Mock-up ကိုကြည့်ပြီး "တော်တော်ခက်မှာပဲ" လို့ ထင်ကောင်းထင်နိုင်ပါတယ်။ စိတ်မပူပါနဲ့၊ ကျွန်တော်တို့ တစ်ဆင့်ချင်းစီ အေးအေးဆေးဆေး သွားကြမှာပါ။ Web development နည်းပညာက အမြဲတမ်း အလျှင်အမြန် ပြောင်းလဲနေတာဖြစ်လို့ ဝါရင့် Developers တွေတောင် Project အသစ်စတိုင်း Tools အသစ်တွေကို ပြန်သင်ယူရလေ့ရှိပါတယ်။ ဒီသင်တန်းမှာ မင်းတို့ တွေ့ရှိလာမှာကတော့ Developer တစ်ယောက်ဖြစ်လာဖို့ဆိုတာ "ဘယ်လိုသင်ယူရမလဲ" ဆိုတာကို သင်ယူခြင်းပဲ ဖြစ်ပါတယ်။

အခု ပထမဆုံးသင်ခန်းစာမှာတော့ အောက်မှာပြထားတဲ့အတိုင်း အခြေခံ Layout ကိုပဲ အရင်ဆုံး တည်ဆောက်မှာ ဖြစ်ပါတယ်။

ဒါကို အောင်အောင်မြင်မြင် လုပ်နိုင်ဖို့ အောက်ပါအဆင့်တွေကို လိုက်နာသွားပါမယ် -

  1. Command line နဲ့ စတင်ခြင်း: Project ရဲ့ File structure ကို တည်ဆောက်ဖို့ အခြေခံ Commands တွေကို လေ့လာပါမယ်။
  2. HTML အနည်းငယ် ထည့်သွင်းခြင်း: Project အတွက် အဓိက HTML file ကို တည်ဆောက်ပြီး အရေးကြီးတဲ့ HTML elements တွေအကြောင်း သင်ယူပါမယ်။
  3. Git ကို အသုံးပြုခြင်း: Git version control tool ရဲ့ အခြေခံတွေကို လေ့လာပြီး ဒီ project အတွက် Repository ဆောက်တာနဲ့ ပထမဆုံး Commits လုပ်တာတွေကို လုပ်ဆောင်ပါမယ်။
  4. စာမျက်နှာကို Outline ဆွဲခြင်း: HTML အကြောင်းကို ပိုနက်နက်ရှိုင်းရှိုင်း လေ့လာပြီး Project အတွက် အခြေခံ Structure ကို စီစဉ်ပါမယ်။
  5. Git workflow ကို ပြန်လည်သုံးသပ်ခြင်း: လုပ်ဆောင်ထားတဲ့ တိုးတက်မှုတွေကို local Git repository ထဲမှာ Commit လုပ်ပါမယ်။
  6. GitHub နဲ့ Publish လုပ်ခြင်း: ကိုယ်တိုင်လုပ်ထားတဲ့ Local repository ကို GitHub Pages ပေါ် တင်ပါမယ်။ ဒါဆိုရင် ကမ္ဘာပေါ်က ဘယ်သူမဆို မင်းရဲ့ website ကို ကြည့်လို့ရသွားပါပြီ။

ကဲ... အခုပဲ Coding စတင်လိုက်ရအောင်!

Get Started with the Command Line

အရင်တုန်းကတော့ ကွန်ပျူတာကို စာသားတွေရိုက်ပြီး ခိုင်းရတဲ့ Command-line interface (CLI) ဒါမှမဟုတ် Terminal နဲ့ပဲ အသုံးပြုခဲ့ကြတာပါ။ အခုခေတ်မှာ Windows နဲ့ macOS တို့လို ပုံတွေ၊ ခလုတ်တွေနဲ့ သုံးရတဲ့ Operating Systems တွေ ပေါ်လာတော့ CLI ကို သုံးစရာမလိုတော့သလို ဖြစ်နေပေမဲ့ Developers တွေကတော့ အလုပ်တော်တော်များများအတွက် CLI ကို အခုထိ သုံးနေရတုန်းပါပဲ။

အခုလောလောဆယ်တော့ ကျွန်တော်တို့ Project ကို Set up လုပ်ဖို့ CLI ကို သုံးသွားပါမယ်။ အဲ့ဒီမတိုင်ခင် ဒါကို ဘယ်လိုဖွင့်ရမလဲဆိုတာ အရင်ကြည့်ရအောင်။

Access the Command Line (Command Line ကို ဖွင့်နည်း)

macOS အသုံးပြုသူများ

macOS မှာ Terminal ဆိုတဲ့ App ပါပြီးသားပါ။ Spotlight Search (စာရိုက်ပြီး ရှာဖွေခြင်း) မှာ "terminal" လို့ ရိုက်ရှာပြီး ဖွင့်နိုင်ပါတယ်။ Spotlight Search ကို ဖွင့်ဖို့ မှန်ဘီလူးပုံလေးကို နှိပ်ပါ ဒါမှမဟုတ် Command+Space ကို နှိပ်ပါ။

Windows အသုံးပြုသူများ

Prework မှာ Git Bash ကို Install လုပ်ခဲ့ပြီးပြီနော်။ Windows စခရင်ရဲ့ အောက်ခြေ ဘယ်ဘက်မှာရှိတဲ့ Search box မှာ "git bash" လို့ ရိုက်ရှာပြီး ဖွင့်လိုက်ပါ။ ဒါမှမဟုတ် Windows+S ကို နှိပ်ပြီး ရှာဖွေနိုင်ပါတယ်။

Use the Command Line (စတင်အသုံးပြုခြင်း)

Command Line ကို ဖွင့်လိုက်တဲ့အခါ စာရိုက်ဖို့ စောင့်နေတဲ့ cursor လေး ပါတဲ့ window တစ်ခု ပေါ်လာပါလိမ့်မယ်။ အဲ့ဒီ window ထဲမှာ pwd လို့ ရိုက်ပြီး Enter ခေါက်လိုက်ပါ။

pwd ဆိုတာ "Print Working Directory" ကို ပြောတာပါ။ Print ဆိုတာက စခရင်ပေါ်မှာ ပြပေးဖို့ဖြစ်ပြီး Working Directory ဆိုတာကတော့ CLI က အခုလက်ရှိ ရောက်နေတဲ့ Folder ကို ပြောတာပါ။ ဒါကြောင့် pwd လို့ ရိုက်တာဟာ "ငါအခု ဘယ်နေရာ ရောက်နေတာလဲ?" လို့ မေးလိုက်တာပါပဲ။

IMPORTANT

ဒီ boot camp မှာ အောင်မြင်ဖို့အတွက် code တွေကို copy/paste မလုပ်ဘဲ တစ်ကြောင်းချင်းစီ ကိုယ်တိုင် လက်နဲ့ ရိုက်ရေးပါ။ ဒါမှသာ ကျွမ်းကျင်တဲ့ coder တစ်ယောက်ဖြစ်ဖို့ လိုအပ်တဲ့ muscle memory ကို အမြန်ဆုံး တည်ဆောက်နိုင်မှာပါ။

Terminal ဒါမှမဟုတ် Git Bash ကို စဖွင့်လိုက်ရင် ပုံမှန်အားဖြင့် ကိုယ့်ရဲ့ User Directory (မိမိအကောင့်အမည်နဲ့ folder) ထဲမှာ ရောက်နေပါလိမ့်မယ်။ pwd လို့ ရိုက်ကြည့်ရင် Windows မှာ /c/Users/your-username လို့ ပြပြီး macOS မှာတော့ /Users/your-username လို့ ပြပါလိမ့်မယ်။

အခု ဒီ Directory ထဲမှာ ဘာတွေရှိလဲဆိုတာ ls လို့ ရိုက်ပြီး ကြည့်ရအောင်။ ဒီ Command က လက်ရှိရောက်နေတဲ့ နေရာက Files တွေနဲ့ Folders တွေကို စာရင်းပြပေးတာပါ။ pwd က "ငါဘယ်မှာလဲ?" လို့ မေးတာဆိုရင် ls က "ဒီမှာ ဘာတွေရှိလဲ?" လို့ မေးတာပါပဲ။

ls ရိုက်လိုက်ရင် Documents, Pictures, Music နဲ့ Desktop စတဲ့ folder တွေကို တွေ့ရပါလိမ့်မယ်။

Nerd Note

ကွန်ပျူတာ စဖွင့်လိုက်ရင် မြင်ရတဲ့ Desktop ဆိုတာ တကယ်တော့ မင်းရဲ့ User Directory ထဲမှာ ရှိတဲ့ Desktop ဆိုတဲ့ Folder လေးပါပဲ။

အခု Desktop ပေါ်ကို CLI ကနေ သွားကြည့်ရအောင်။ အောက်က Commands တွေကို တစ်ကြောင်းချင်း ရိုက်ပြီး Enter ခေါက်ပါ - 

        cd Desktop
        pwd

PAUSE

cd ဆိုတာ ဘာကို ဆိုလိုတာလို့ ထင်လဲ?

အဖြေ - cd ဆိုတာ Change Directory ကို ပြောတာပါ။ Folder တစ်ခုကနေ တစ်ခုကို ကူးပြောင်းဖို့ သုံးပါတယ်။
Show Answer

အခု အောက်ကအတိုင်း ထပ်ရိုက်ကြည့်ပါ။ pwd မှာ ဘာတွေပြလဲဆိုတာ သတိထားကြည့်နော် - 

        cd ..
        pwd
        cd Desktop
        pwd

ဒါကြောင့် cd .. လို့ ရိုက်ရင် အနောက်ကို တစ်ဆင့် (ဒါမှမဟုတ် အပေါ်က Directory ထဲကို) ပြန်သွားတယ်ဆိုတာ မှတ်ထားလိုက်ပါ!

အခု Desktop ပေါ် ပြန်ရောက်နေပြီဆိုရင် ls လို့ ရိုက်ကြည့်ပါ။ ကိုယ့် Desktop ပေါ်မှာ တကယ်ရှိနေတဲ့ ဖိုင်တွေကို မြင်ရတယ်မလား?

အခု Folder အသစ်တစ်ခု ဆောက်ကြည့်ရအောင်။ အောက်ပါအတိုင်း ရိုက်ပါ - 

        mkdir projects
        ls

mkdir ဆိုတာ "Make Directory" ပါ။ Folder အသစ်ဆောက်တာကို ဆိုလိုပါတယ်။ projects ဆိုတဲ့ folder လေး ဆောက်လိုက်တာပါ။

Introducing Git

Codebases တွေကို စနစ်တကျ စီမံခန့်ခွဲဖို့အတွက် version control အကြောင်း သိထားဖို့ လိုပါတယ်။ Git ဟာ ကမ္ဘာပေါ်မှာ အသုံးအများဆုံး version control system တစ်ခုပါပဲ။

Git က မင်းရဲ့ အလုပ်တွေကို "Save Points" အနေနဲ့ မှတ်ထားပေးနိုင်ပါတယ်။ ဒါကို commits လို့ ခေါ်ပါတယ်။ အလုပ်တစ်ခုကို အပိုင်းလိုက် ပြီးသွားတိုင်း commit လုပ်ထားတဲ့ အလေ့အကျင့်က အရမ်းကောင်းပါတယ်။ ဒါမှသာ အပြောင်းအလဲ မှတ်တမ်း (history) ရှိနေမှာဖြစ်ပြီး လိုအပ်ရင် အရင်ဗားရှင်း အဟောင်းဆီကို ပြန်သွား (revert) လို့ ရမှာပါ။

ON THE JOB

Software developer တစ်ယောက်အနေနဲ့ production ဆိုတဲ့ စကားလုံးကို ခဏခဏ ကြားရပါလိမ့်မယ်။ ဒါကတော့ users တွေ လက်ရှိ တကယ်သုံးနေတဲ့ live version (လက်ရှိသုံးနေတဲ့ website/app) ကို ပြောတာပါ။ တစ်ခါတလေမှာ ကုဒ်အမှားတွေ production ပေါ် ပါသွားတတ်ပါတယ်။ အဲ့ဒီလိုဖြစ်ရင် Git ကို သုံးပြီး အရင် အလုပ်လုပ်နေတဲ့ ဗားရှင်းအဟောင်းဆီ ချက်ချင်း ပြန်ပြောင်းလိုက်လို့ ရပါတယ်။

Git ကို သုံးရတဲ့ တခြား အကျိုးကျေးဇူးတွေကတော့ -

  • Remote location (အင်တာနက်ပေါ်က သိမ်းဆည်းရာနေရာ) ဆီ commits တွေကို push လုပ်ထားနိုင်ပါတယ်။ ဒါဆိုရင် မင်းရဲ့ ကွန်ပျူတာ ပျက်သွားရင်တောင် code တွေ မပျောက်တော့ပါဘူး။ စက်တစ်လုံးကနေ တစ်လုံး ပြောင်းသုံးရတာလည်း လွယ်သွားပါတယ်။
  • အဖွဲ့လိုက် (team) အလုပ်လုပ်တဲ့အခါမှာလည်း အရမ်းအသုံးဝင်ပါတယ်။ Git သာ မရှိရင် developer အချင်းချင်း code တွေ မျှဝေတဲ့အခါ တစ်ယောက်ရေးတာကို တစ်ယောက်က မတော်တဆ ဖျက်မိတာမျိုး၊ ထပ်ရေးမိတာမျိုးတွေ ဖြစ်ပြီး တိုင်ပတ်နေပါလိမ့်မယ်။
  • Git ကို သုံးပြီး codebase တစ်ခုတည်းကနေ ဗားရှင်းခွဲ (branches) တွေ ခွဲထုတ်ပြီး အလုပ်လုပ်လို့ ရပါတယ်။ အားလုံးပြီးလို့ ပြန်ပေါင်း (merge) တဲ့အခါ code တွေ ထပ်နေရင်လည်း Git က conflicts အနေနဲ့ သတိပေးမှာ ဖြစ်လို့ ပြန်ပြင်ဖို့ အခွင့်အရေး ရပါတယ်။

Create a Run Buddy Git Repository

အခု Run Buddy folder ကို Git repository အဖြစ် ပြောင်းကြရအောင်။ ဒါဟာ ရိုးရိုး folder တစ်ခုကို version control (ဗားရှင်း ထိန်းချုပ်မှု) လုပ်နိုင်စွမ်းရှိတဲ့ folder အဖြစ် ပြောင်းလဲလိုက်တာပါ။

IMPORTANT

ပုံမှန်အားဖြင့် local Git မှာ default branch နာမည်က master ဖြစ်နေတတ်ပါတယ်။ ဒါပေမဲ့ GitHub က အခုနောက်ပိုင်း main ကိုပဲ အသုံးပြုတော့တာကြောင့် ကျွန်တော်တို့ စက်ထဲက Git setting ကိုလည်း main ဖြစ်အောင် ပြင်ပေးဖို့ လိုပါတယ်။

macOS သုံးတဲ့သူတွေအနေနဲ့ Git version 2.28 နဲ့အထက် ဖြစ်ဖို့ လိုပါတယ်။ ဒါကြောင့် အောက်က command ကို အရင်ရိုက်ပြီး update လုပ်ပေးပါ (Windows သုံးသူတွေ လုပ်စရာမလိုပါ) -

# macOS သုံးသူများသာ ရိုက်ရန်
        brew install git

macOS ရော Windows အသုံးပြုသူပါ အောက်က command ကို ရိုက်ပြီး default branch ကို main အဖြစ် သတ်မှတ်လိုက်ပါ -

git config --global init.defaultBranch main

ဒီ command ရိုက်လို့ ဘာစာမှ ထပ်မတက်လာဘဲ အောက်ကြောင်းဆင်းသွားရင် အောင်မြင်သွားပါပြီ။

အခု command line မှာ run-buddy folder ထဲ ရောက်နေပြီဆိုရင် အောက်ကအတိုင်း ရိုက်လိုက်ပါ -

git init

ဒါဆိုရင် "Initialized empty Git repository" ဆိုတဲ့ စာသားလေး ပေါ်လာပါလိမ့်မယ်။ အကယ်၍ git ဆိုတဲ့ စကားလုံးကို မသိဘူးလို့ ပြနေရင်တော့ Git website ကနေ Git ကို install လုပ်ထားဖို့ လိုပါမယ်။

ဘာမှ ထူးမခြားနားဘူးလို့ ထင်ရပေမဲ့ တကယ်တော့ run-buddy folder ထဲမှာ .git ဆိုတဲ့ hidden folder (ဖျောက်ထားတဲ့ ဖိုင်တွဲ) လေးတစ်ခု ဆောက်လိုက်တာပါ။ ဒါကို မျက်စိနဲ့ မြင်ချင်ရင်တော့ အောက်က command ကို ရိုက်ကြည့်ပါ -

ls -a

ဒီ ls -a က ဖျောက်ထားတဲ့ ဖိုင်တွေပါ အကုန်ပြပေးတာပါ။ ဖိုင်နာမည် ရှေ့မှာ . (အစက်) ပါရင် အဲ့ဒါ hidden file ပါ။ ဒီ .git folder က repository တစ်ခုလုံးရဲ့ အချက်အလက်တွေကို သိမ်းထားတာဖြစ်လို့ ကျွန်တော်တို့ သွားမပြင်တာ အကောင်းဆုံးပါပဲ။

Try Some Git Commands

Git repository ဆောက်ပြီးပြီဆိုရင် ကျွန်တော်တို့ရဲ့ Code တွေ ပြောင်းလဲမှုကို စတင်စောင့်ကြည့် (track) လို့ရပါပြီ။ အရင်ဆုံး git status ဆိုတဲ့ command ကို ရိုက်ကြည့်ပါ။ အောက်ပါအတိုင်း တွေ့ရပါလိမ့်မယ် -

IMPORTANT

macOS သုံးတဲ့သူတွေအနေနဲ့ .DS_Store ဆိုတဲ့ဖိုင်ကိုပါ တွေ့နေရရင် လောလောဆယ် လျစ်လျူရှုထားလိုက်ပါ။ ဒါကို နောက်ပိုင်းမှာ ဘယ်လိုဖျောက်ရမလဲဆိုတာ သင်ပေးပါမယ်။

git status ဆိုတာ နောက်ဆုံးအကြိမ် Commit လုပ် (သိမ်းဆည်း) ပြီးကတည်းက ဘယ်ဖိုင်တွေ ပြောင်းလဲသွားလဲဆိုတာကို အမြန်စစ်ဆေးပေးတဲ့ command ပါ။ အခုဆိုရင် Git က index.html ကို Untracked files (စောင့်ကြည့်ခြင်းမခံရသေးတဲ့ဖိုင်) စာရင်းထဲမှာ ပြနေပါလိမ့်မယ်။ Git ရဲ့ သဘောက ကျွန်တော်တို့က "စောင့်ကြည့်ပေးပါ" လို့ ပြောတဲ့ဖိုင်ကိုပဲ သူက ဂရုစိုက်တာပါ။

ကျွန်တော်တို့က index.html ကို Git ထဲ ထည့်ချင်တာဖြစ်လို့ အောက်ပါအတိုင်း ရိုက်ထည့်လိုက်ပါ -

git add index.html

      git status

git add ဆိုတာကတော့ ဖိုင်တွေကို Staging (စင်မြင့်ထက်) ပေါ်တင်ပေးလိုက်တာပါ။ ဇာတ်ပွဲမစခင် မင်းသား၊ မင်းသမီးတွေကို စင်နောက်မှာ အသင့်ပြင်ခိုင်းထားတဲ့ ပုံစံမျိုးပေါ့။ Git အခေါ်အဝေါ်အရဆိုရင်တော့ ကျွန်တော်တို့က Commit (တကယ်မသိမ်းရသေးဘူး)၊ ဒါပေမဲ့ ဘာတွေကို သိမ်းမယ်ဆိုတာ Git ကို အသိပေးလိုက်တာ ဖြစ်ပါတယ်။

IMPORTANT

git add လုပ်လိုက်တဲ့အခါ ဘာစာသားမှ ပေါ်လာမှာမဟုတ်ဘဲ နောက်ထပ် command ရိုက်ဖို့ နေရာပဲ ပြန်ပေါ်လာပါလိမ့်မယ်။ အမှားတစ်ခုခုရှိမှသာ Git က စာပြမှာဖြစ်လို့ ဘာမှမပြရင် အောင်မြင်တယ်လို့ မှတ်ယူနိုင်ပါတယ်။

Make Your First Git Commit

အခု git status ကို ပြန်ရိုက်ကြည့်ရင် index.html ဟာ "Changes to be committed" (သိမ်းဆည်းဖို့ အဆင်သင့်ဖြစ်နေတဲ့ ပြောင်းလဲမှုများ) ဆိုတဲ့ စာရင်းထဲကို ရောက်သွားတာ တွေ့ရပါမယ်။

ဒါကို တကယ် သိမ်းဆည်း (Commit) ဖို့အတွက် အောက်ပါ Command ကို ရိုက်ထည့်လိုက်ပါ -

git commit -m "page template"

git commit ဆိုတာ Staging ထဲမှာ ရှိသမျှ အရာအားလုံးကို တကယ် သိမ်းဆည်းလိုက်တာ ဖြစ်ပါတယ်။ -m "message" အပိုင်းကတော့ ဒီတစ်ခါ သိမ်းလိုက်တဲ့ Code ထဲမှာ ဘာတွေ ပြင်ဆင်ထားသလဲဆိုတာကို အကျဉ်းချုပ် ရေးပေးတာ ဖြစ်ပါတယ်။

SHOW PRO TIP
HIDE PRO TIP

Commit တိုင်းမှာ ဘာတွေပြင်လိုက်လဲဆိုတာ ရှင်းပြဖို့ Message ပါရပါမယ်။ အကယ်လို့ -m "..." အပိုင်းကို ထည့်ဖို့ မေ့သွားရင် ထွက်ရခက်တဲ့ Prompt (စာရိုက်မျက်နှာပြင်) ကြီးထဲ ရောက်သွားတတ်လို့ သတိထားပါ!

ပထမဆုံးအကြိမ် Commit လုပ်တာဆိုရင် Git က "မင်း ဘယ်သူလဲ?" (Identify yourself) ဆိုပြီး အောက်ပါအတိုင်း နာမည်နဲ့ Email မေးပါလိမ့်မယ် -

git config --global user.name "Your Name"
git config --global user.email "you@example.com"

ဒါက ပုံမှန်ပါပဲ။ Terminal က ခိုင်းတဲ့အတိုင်း ကိုယ့်နာမည်နဲ့ Email ကို အရင် သတ်မှတ်ပေးလိုက်ပါ။ ပြီးရင်တော့ git commit -m "page template" ကို ပြန်ရိုက်ပါ။

Terminal ထဲမှာ "1 file changed, 11 insertions" ဆိုတဲ့ စာသားလေး ပေါ်လာပြီဆိုရင် မင်းရဲ့ ပထမဆုံး Commit ကို အောင်အောင်မြင်မြင် လုပ်ဆောင်နိုင်ခဲ့ပါပြီ! အခု လုပ်ငန်းစဉ်ကို နောက်တစ်ခါ ထပ်ပြီး လေ့ကျင့်ကြည့်ရအောင်။

Commit နောက်တစ်ခု ထပ်လုပ်ကြည့်ရအောင်

အခု index.html ဖိုင်ထဲက <body> Element ထဲမှာ အောက်ကစာသားလေးတွေကို ပြင်ပြီး ရိုက်ထည့်လိုက်ပါ -

        <body>

            <h1>RUN BUDDY</h1>
            What We Do What You Do Your Trainers Reach Out
        </body>

ပြီးရင် git status ကို ပြန်ရိုက်ကြည့်ပါ။ index.html ဖိုင်ကို Modified (ပြင်ဆင်ထားသည်) ဆိုပြီး ပြနေပါလိမ့်မယ်။ ဒါပေမဲ့ ဒါဟာ Staging ထဲ မရောက်သေးတဲ့အတွက် အောက်ပါအတိုင်း အဆင့်ဆင့် ပြန်လုပ်ပေးရပါမယ် -

git add index.html
    git commit -m "add more link text"

အခုဆိုရင် ကျွန်တော်တို့ Commit ၂ ခု လုပ်ပြီးသွားပါပြီ။ ကိုယ်လုပ်ခဲ့သမျှ မှတ်တမ်းတွေကို ပြန်ကြည့်ချင်ရင် git log ဆိုတဲ့ Command ကို ရိုက်ကြည့်ပါ။ ဘယ်သူ၊ ဘယ်နေ့၊ ဘာ Message နဲ့ Commit လုပ်ခဲ့လဲဆိုတာ အကုန်တွေ့ရပါလိမ့်မယ်။

IMPORTANT

git log ကြည့်နေတဲ့ Window ထဲကနေ ပြန်ထွက်ချင်ရင် ကိုယ့် Keyboard က q အက္ခရာလေးကို နှိပ်လိုက်ရုံပါပဲ။

နောက်ပိုင်း တခြား Developers တွေနဲ့ စုပေါင်းပြီး App အကြီးကြီးတွေ ရေးတဲ့အခါ ဒီ Git ဟာ ဘယ်လောက်တောင် အသုံးဝင်မလဲဆိုတာ စဉ်းစားကြည့်ပါ။ တကယ်ကို အားကိုးရတဲ့ Tool တစ်ခုပါပဲ!

စာမျက်နှာကို Outline ဆွဲခြင်း

Git အကြောင်းကို နောက်ပိုင်းမှာ ထပ်ပြီး လေ့ကျင့်ဖို့ အခွင့်အရေး ရဦးမှာပါ။ အခုတော့ HTML တည်ဆောက်တဲ့အပိုင်းကို ပြန်သွားကြရအောင်။

Run Buddy က ပေးထားတဲ့ landing page ရဲ့ design mock-up (ဒီဇိုင်းပုံစံကြမ်း) ကို မှတ်မိသေးတယ်မလား? အောက်ကပုံမှာ ပြထားတဲ့ mock-up ကို တစ်ခါလောက် ပြန်ကြည့်ပြီး သေချာ လေ့လာကြည့်လိုက်ပါ -

Coding တန်းမရေးခင်မှာ ကိုယ်ဘာတွေ ရေးရမလဲဆိုတာကို Outline အရင်ဆွဲထားတာက အချိန်ကုန်သက်သာစေသလို အမှားလည်း နည်းစေပါတယ်။

HTML ကို သေတ္တာ (Boxes) တွေလို စဉ်းစားခြင်း

အရောင်စုံ၊ ပုံစံစုံနဲ့ ရှိနေတဲ့ Design တစ်ခုကို raw HTML အဖြစ် ပြောင်းဖို့ဆိုတာ အစပိုင်းမှာ တုန်လှုပ်စရာ ကောင်းနိုင်ပါတယ်။ ဒါကြောင့် Coding တန်းမရေးခင် စာမျက်နှာကို Outline (ပုံကြမ်း) အရင်ဆွဲဖို့က အရမ်းအရေးကြီးပါတယ်။

Developer တွေအနေနဲ့ အစီအစဉ်မရှိဘဲ Coding တန်းရေးတာကို ရှောင်ရပါမယ်။ HTML ရဲ့ သဘောတရားက တကယ်တော့ တစ်ခုထဲမှာ တစ်ခုထပ်နေတဲ့ ဒါမှမဟုတ် ဘေးချင်းယှဉ်နေတဲ့ Boxes (သေတ္တာ) တွေနဲ့ တူပါတယ်။

အပေါ်ကပုံထဲမှာ

elements တွေကို သုံးထားတာ တွေ့ရပါမယ် -

  • ဆိုတာ "Content Division" လို့ အဓိပ္ပာယ်ရပြီး သက်ဆိုင်ရာ အချက်အလက်တွေကို စုစည်းထားတဲ့ Container (ပုံး) တစ်ခုလိုပါပဲ။
  • Box 1 နဲ့ Box 2 က တစ်ခုပေါ်တစ်ခု ထပ်မနေပါဘူး၊ သီးခြားစီ ရှိနေတဲ့ Content တွေပါ။ ဥပမာ- Box 1 က ကြော်ငြာဖြစ်ပြီး Box 2 က Navigation bar (မီနူးဘား) ဖြစ်နိုင်ပါတယ်။
  • ဒီ Boxes တွေထဲမှာ နောက်ထပ် Boxes တွေ ထပ်ထည့်လို့ရပါတယ်။ ဥပမာ- Box 3 ဆိုတဲ့ အစုအဝေးထဲမှာ Box 4 နဲ့ 5 ဆိုတဲ့ Links လေးတွေကို ထည့်ထားတာမျိုးပါ။

Box 3 ကို မသုံးဘဲ Box 4 နဲ့ 5 ကို Box 2 ထဲ တန်းထည့်လို့ မရဘူးလားဆိုရင် ရပါတယ်။ ဒါပေမဲ့ အုပ်စုတူတဲ့ Content တွေကို စုစည်းထားခြင်းက Code ကို ပိုပြီး သေသပ်စေသလို ရှုပ်ထွေးတဲ့ Information တွေကို စနစ်တကျ ဖြစ်စေပါတယ်။

PAUSE

ဘာကြောင့် nested <div> တွေကို indent လုပ်ဖို့ လိုတာလဲ?

nested <div> (တစ်ခုထဲမှာ တစ်ခု ထပ်နေတဲ့ tag) တွေကို indent လုပ်ပြီး (အထဲကို နည်းနည်းဝင်ပြီး) ရေးရတာဟာ code တွေကို ပြန်ဖတ်တဲ့အခါ ပိုပြီး readable ဖြစ်အောင် (မျက်စိထဲမှာ ရှင်းရှင်းလင်းလင်း မြင်သာအောင်) လို့ ဖြစ်ပါတယ်။

Show Answer

Layout ကို အပိုင်းလိုက် ခွဲခြားခြင်း

ကျွန်တော်တို့ Mock-up ပုံကို ကြည့်ပြီး Logical boxes ဒါမှမဟုတ် Containers တွေအဖြစ် ဘယ်လို ခွဲထုတ်မလဲဆိုတာ စဉ်းစားကြည့်ရအောင်။ တချို့ Containers တွေက တခြား Boxes တွေကို စုစည်းပေးဖို့သက်သက်ပဲ သုံးတာမျိုးလည်း ရှိနိုင်ပါတယ်။ ဒါကို အခုချက်ချင်း နားမလည်သေးရင်လည်း စိတ်မပူပါနဲ့၊ Apps တွေ အများကြီး ဆောက်ကြည့်လေလေ Layout ချရတာ ပိုကျွမ်းကျင်လာလေလေ ဖြစ်မှာပါ။

လက်တွေ့ လေ့ကျင့်ခန်း

အခုပဲ ခဲတံနဲ့ စက္ကူတစ်ရွက်ယူပြီး Mock-up ထဲက ကျန်တဲ့ အပိုင်းတွေကို ကိုယ်တိုင် Outline ဆွဲကြည့်ပါ။ ကွန်ပျူတာရှေ့က ခဏဖယ်ပြီး လက်နဲ့တကယ် ဆွဲကြည့်တာက ပြဿနာတွေကို ရှုထောင့်အသစ်ကနေ မြင်နိုင်ဖို့ အများကြီး အထောက်အကူပြုပါတယ်။

ON THE JOB

လုပ်ငန်းခွင်မှာ Team တွေဟာ Code တစ်ကြောင်းမှ မရေးခင် Project တစ်ခုကို ရက်ပေါင်းများစွာ ဒါမှမဟုတ် သီတင်းပတ်နဲ့ချီပြီး စီစဉ် (Planning) လေ့ရှိပါတယ်။ စက္ကူပေါ်မှာ ပုံဆွဲတာဟာ အချိန်ဖြုန်းတာ မဟုတ်ဘဲ ရေရှည်မှာ အချိန်ကုန်သက်သာစေသလို ပြဿနာတွေကိုလည်း ကြိုတင်ကာကွယ်နိုင်ပါတယ်။

ဆွဲထားတဲ့ Outline ကို အနားမှာ အသင့်ထားပါ။ ကျွန်တော်တို့ ရေးမယ့် Code နဲ့ တစ်ထပ်တည်း မကျရင်လည်း ပြဿနာမရှိပါဘူး။ Webpage တစ်ခုကို ဆောက်ဖို့ နည်းလမ်းတွေ အများကြီး ရှိလို့ပါ။

<div> Element ရဲ့ အားနည်းချက်

ကျွန်တော်တို့ <div> အကြောင်း သိထားပြီးပြီဆိုတော့ index.html ကို အောက်ကပုံစံအတိုင်း ပြင်ရေးချင်စိတ် ပေါက်လာနိုင်ပါတယ် -

<body>
  <div>
    <h1>RUN BUDDY</h1>
  </div>
</body>

ဒါက အစကောင်းတစ်ခုပါ။ ကုမ္ပဏီနာမည်နဲ့ Navigation links တွေဟာ <div> ဆိုတဲ့ Box ထဲမှာ ရှိမယ်ဆိုတာ သိနိုင်ပါတယ်။ ဒါပေမဲ့ <div> ရဲ့ ပြဿနာက သူက အရမ်း Generic ဖြစ်လွန်းတာ (တိမ်လွန်းတာ) ပါပဲ။

"Div Soup" ဆိုတာ ဘာလဲ?

အကယ်လို့ ကျွန်တော်တို့ရဲ့ Layout က အရမ်းရှုပ်ထွေးပြီး နေရာတိုင်းမှာ <div> တွေပဲ သုံးထားမယ်ဆိုရင် ဘယ်ဟာက ဘာအတွက်လဲဆိုတာ တစ်ချက်ကြည့်ရုံနဲ့ သိဖို့ အရမ်းခက်သွားပါလိမ့်မယ်။ အောက်က ဥပမာကို ကြည့်ပါ -

  <div>
    <div>Search</div>
    <div>
      <div>John Smith</div>
      <div>Jane Doe</div>
    </div>
  </div>
  <div>

HTML5 မှာ <div> တွေချည်းပဲ သုံးမယ့်အစား ပိုပြီး အဓိပ္ပာယ်ပေါ်လွင်စေတဲ့ Semantic elements တွေကို မိတ်ဆက်ပေးခဲ့ပါတယ်။ အောက်က Code ကို ပြန်ပြင်ရေးထားတာ ကြည့်ကြည့်ပါ -

ပိုပြီး အသေးစိတ်လေ့လာချင်ရင် HTML5 နဲ့ ပတ်သက်တဲ့ MDN Web Docs on HTML5 မှာ သွားရောက်ဖတ်ရှုနိုင်ပါတယ်။

ဒီ Code အပိုင်းအစက ဘာကို ဆိုလိုတာလဲဆိုတာ ဇဝေဇဝါ ဖြစ်စရာ လိုသေးလား? မလိုတော့ပါဘူး! ဒါဟာ Website ရဲ့ ထိပ်ပိုင်း (Header) ဖြစ်တယ်ဆိုတာ <header> element ကို ကြည့်တာနဲ့ တန်းသိနိုင်ပါတယ်။

နည်းပညာပိုင်းအရ ပြောရရင် <header> ဟာ <div> နဲ့ အလုပ်လုပ်ပုံ အတူတူပါပဲ။ ဒါပေမဲ့ သူ့ကို သုံးရတဲ့ ရည်ရွယ်ချက်က Code ဖတ်မယ့် Developers တွေတင်မကဘဲ Search Engines (Google လိုမျိုး) ကပါ Website ရဲ့ ဘယ်အပိုင်းက ပိုအရေးကြီးလဲဆိုတာကို လွယ်လွယ်ကူကူ ခွဲခြားသိမြင်နိုင်စေဖို့ ဖြစ်ပါတယ်။

ကျွန်တော်တို့ Layout ရဲ့ အောက်ဆုံးအပိုင်းမှာလည်း သိသိသာသာ ကွဲပြားတဲ့ Footer အပိုင်း ရှိပါတယ်။ အဲ့ဒါအတွက်လည်း Semantic element ရှိပါတယ် -

တကယ်လို့ ကိုယ်သုံးချင်တဲ့ အပိုင်းအတွက် ကိုက်ညီမယ့် Semantic elements ရှာမတွေ့ဘူးဆိုရင်တော့ <div> ကိုပဲ ပြန်သုံးလို့ ရပါတယ်။ ဒါပေမဲ့ အခုလောလောဆယ် ကျွန်တော်တို့ Website ရဲ့ အပိုင်းတွေကို ခွဲခြားဖို့ <section> element ကို သုံးကြည့်ရအောင်။

ဒီသင်ခန်းစာမှာ ပါတဲ့ Code တွေကို Copy-Paste မလုပ်ပါနဲ့! ကိုယ်တိုင် လက်နဲ့ လိုက်ရိုက်ဖို့ ထပ်ပြီး သတိပေးပါရစေ။

ဒီမှာ အသစ်တစ်ခု တွေ့ပါလိမ့်မယ်။ အဲ့ဒါကတော့ <!-- --> ပါ။ ဒါကို Code Comment လို့ ခေါ်ပါတယ်။ ဒါတွေဟာ Browser မှာ ပေါ်မှာ မဟုတ်ဘဲ Developer တွေ Code ကို ပြန်ဖတ်တဲ့အခါ မှတ်သားရလွယ်အောင် မှတ်စုရေးတဲ့ နေရာဖြစ်ပါတယ်။

အခု Save လိုက်ပါ၊ ပြီးရင် Browser ကို Refresh လုပ်ပြီး ကြည့်လိုက်ပါ။ Website က အဖြူရောင်နောက်ခံမှာ စာသားတွေပဲ ရှိနေဦးမှာမို့လို့ သိပ်ပြီးတော့ မလှသေးပါဘူး။ ဒါပေမဲ့ မစိုးရိမ်ပါနဲ့၊ ကျွန်တော်တို့ အရေးကြီးတဲ့ အခြေခံ (Groundwork) ကို ချလိုက်တာဖြစ်လို့ နောက်ထပ်အဆင့်တွေမှာ ပြင်ရတာ ပိုလွယ်သွားပါလိမ့်မယ်။

Save Your Progress with Git

အလုပ်လုပ်တဲ့ Workflow ထဲမှာ Git ကို ခဏခဏ Commit လုပ်တဲ့ အလေ့အကျင့် ရှိထားဖို့ လိုပါတယ်။ ဒါမှ ကိုယ့်ရဲ့ အလုပ်တွေ မတော်တဆ မပျောက်ပျက်မှာဖြစ်သလို၊ လိုအပ်ရင် အရင်က သိမ်းခဲ့တဲ့ "Save point" ကိုလည်း ပြန်သွားနိုင်မှာပါ။ အခုဆိုရင် Website ရဲ့ အခြေခံ Structure ကို အကြမ်းဖျင်း ဆွဲပြီးပြီဖြစ်လို့ Commit လုပ်ဖို့ အကောင်းဆုံး အချိန်ပါပဲ။

.gitignore ဖိုင် ဖန်တီးခြင်း

Commit မလုပ်ခင်မှာ Project ထဲမှာ ဖိုင်အသစ်တစ်ခု အရင်ဆောက်ရအောင် -

touch .gitignore

ကျွန်တော်တို့က Git ကို "ဒီဖိုင်တွေကိုတော့ စာရင်းထဲမသွင်းနဲ့ (မမှတ်နဲ့)" လို့ ပြောချင်တဲ့အခါ .gitignore ဖိုင်ကို သုံးပါတယ်။ Developers တွေဟာ တစ်ခါတလေ ဖိုင်တွေအကုန်လုံးကို Automatically commit လုပ်တတ်ကြတယ်။ အဲ့ဒီအခါ မလိုလားအပ်တဲ့ ဖိုင်တွေပါသွားပြီး Project ကို ရှုပ်ထွေးစေတတ်ပါတယ်။

ဥပမာ - macOS မှာဆိုရင် Folder တိုင်းမှာ .DS_Store ဆိုတဲ့ ဖိုင်လေးက အလိုလို ပါနေတတ်ပါတယ်။ ဒါဟာ Git ထဲ ပါသွားရင် အလုပ်လုပ်ရတာ စိတ်အနှောင့်အယှက် ဖြစ်စေပါတယ်။ Windows သုံးတဲ့သူတွေကတော့ အဲ့ဒီပြဿနာကို သိသာအောင် touch .DS_Store ဆိုပြီး ဖိုင်အသစ်တစ်ခု စမ်းဆောက်ကြည့်လို့ ရပါတယ်။

IMPORTANT

မင်းက Windows သုံးတာပဲဖြစ်ဖြစ် .gitignore ထဲမှာ .DS_Store ကို ထည့်ထားတာ အကောင်းဆုံးပါပဲ။ ဘာလို့လဲဆိုတော့ တခြား macOS သုံးတဲ့သူတွေနဲ့ အလုပ်တွဲလုပ်တဲ့အခါ Conflict (ပြဿနာ) မတက်အောင် ကြိုတင်ကာကွယ်ထားတာ ဖြစ်ပါတယ်။

Staging နှင့် Commit လုပ်ခြင်း

အခုဆိုရင် အားလုံးဆီမှာ .gitignore ဖိုင် ရှိနေပြီဆိုတော့ အဲ့ဒီဖိုင်ကို VS Code မှာ ဖွင့်လိုက်ပါ။ ပြီးရင် အထဲမှာ .DS_Store လို့ ရိုက်ထည့်ပြီး Save လိုက်ပါ။ ပြီးရင် Command Line မှာ git status လို့ ပြန်ရိုက်ကြည့်လိုက်ပါ။

အခုကြည့်လိုက်ရင် Git က .DS_Store ဖိုင်ကို စာရင်းထဲမှာ မပြတော့ဘဲ လျစ်လျူရှုလိုက်တာ (ignored) ကို တွေ့ရပါမယ်။ အခု Git က မြင်နေရတဲ့ဖိုင်တွေကတော့ -

  • Modified file: index.html (ကျွန်တော်တို့ Semantic elements တွေ ထည့်ပြီး ပြင်ထားတဲ့ဖိုင်)
  • Untracked file: .gitignore (အခုမှ အသစ်ဆောက်လိုက်တဲ့ဖိုင်)

ဒီဖိုင်နှစ်ခုလုံးကို Staging ထဲ တစ်ခါတည်း ထည့်ဖို့အတွက် အောက်ပါ Command နှစ်ခုထဲက တစ်ခုခုကို သုံးနိုင်ပါတယ် -

      git add .

ဒါမှမဟုတ်

      git add -A

ဘယ်တစ်ခုကို သုံးရမလဲဆိုတာ သိဖို့ သူတို့ရဲ့ ကွာခြားချက်ကို ကြည့်ရအောင် -

  • git add . : အခု လက်ရှိရောက်နေတဲ့ Directory နဲ့ သူ့အောက်မှာရှိတဲ့ Subdirectories ထဲက ပြင်ထားတဲ့ဖိုင်တွေ၊ ဖိုင်အသစ်တွေ အကုန်လုံးကို ထည့်ပေးတာပါ။
  • git add -A : Project တစ်ခုလုံးမှာ ရှိသမျှ အပြောင်းအလဲ အားလုံးကို Stage လုပ်ပေးတာပါ။

ဒီ Command တွေက ဖိုင်အများကြီးကို တစ်ခါတည်း Stage လုပ်ချင်တဲ့အခါ အရမ်းအသုံးဝင်ပါတယ်။ ဒါပေမဲ့ .gitignore ဖိုင်ကို အရင်သေချာ မပြင်ထားဘဲ သုံးမိရင် မလိုလားအပ်တဲ့ ဖိုင်တွေပါ Git ထဲ ပါသွားတတ်လို့ သတိထားရပါမယ်။

အားလုံး Stage လုပ်ပြီးပြီဆိုရင်တော့ အောက်ပါ Command နဲ့ Commit လုပ်လိုက်ရအောင် -

      git commit -m "html outline and gitignore"

Publish with GitHub Pages

အရမ်းကောင်းပါတယ်—မင်းကြိုးစားထားသမျှတွေကို Git ထဲမှာ သိမ်းဆည်းပြီးသွားပါပြီ။ ဒါပေမဲ့ ပြဿနာတစ်ခုရှိတာက ဒီ Commits တွေက မင်းရဲ့ Local Computer (ကိုယ့်စက်) ထဲမှာပဲ ရှိနေတာပါ။ အကယ်လို့ မတော်တဆ ကော်ဖီမှောက်ကျတာပဲဖြစ်ဖြစ်၊ လျှပ်စစ်ရှော့ခ်ဖြစ်တာပဲဖြစ်ဖြစ် ဖြစ်ခဲ့ရင် မင်းလုပ်ထားသမျှတွေ အကုန်ပျောက်သွားပါလိမ့်မယ်။ အဲ့ဒါဆိုရင် Run Buddy ကုမ္ပဏီက စိတ်ဆိုးမှာ သေချာပါတယ်။

ဒါကြောင့် ကိုယ့်ရဲ့ Repository ကို လုံခြုံစိတ်ချရတဲ့ Remote location (အဝေးက သိမ်းဆည်းရာနေရာ) တစ်ခုမှာ သိမ်းထားနိုင်ရင် ပိုမကောင်းဘူးလား? အဲ့ဒီအတွက် GitHub က ကူညီပေးမှာပါ။

GitHub ဆိုတာ Git repositories တွေကို လက်ခံသိမ်းဆည်းပေးတဲ့ Website တစ်ခုပါ။ ဒါဟာ အခမဲ့ဖြစ်ပြီး Developers တွေကြားမှာ အရမ်းခေတ်စားတာကြောင့် အကောင့်တစ်ခုရှိနေဖို့ လိုပါတယ်။

IMPORTANT

လူအများစု မှားတတ်တာက Git နဲ့ GitHub ကို အတူတူပဲလို့ ထင်နေကြတာပါ။ Git ဆိုတာက မင်းရဲ့ ကွန်ပျူတာထဲမှာ Install လုပ်ထားတဲ့ Version Control Software ဖြစ်ပြီး GitHub ကတော့ အဲ့ဒီ Repositories တွေကို အင်တာနက်ပေါ်မှာ တင်ထားလို့ရတဲ့ ဝန်ဆောင်မှု (Service) ဖြစ်ပါတယ်။ GitHub အပြင် တခြား ဝန်ဆောင်မှုတွေလည်း ရှိပါသေးတယ်။

ကဲ... အခု ကျွန်တော်တို့ ဆောက်ထားတဲ့ Run Buddy webpage ကို GitHub ပေါ် ဘယ်လိုတင်မလဲ ကြည့်ရအောင်။ ပထမဆုံး အဆင့်အနေနဲ့ GitHub ပေါ်မှာ Remote repository အသစ်တစ်ခု အရင်ဆောက်ရပါမယ်။ ပြီးမှ ကိုယ့်စက်ထဲက Project နဲ့ ချိတ်ဆက်မှာပါ။

GitHub ကို Login ဝင်ပြီးရင် အောက်ပါပုံမှာ ပြထားတဲ့အတိုင်း "New" ဆိုတဲ့ ခလုတ်လေးကို နှိပ်လိုက်ပါ။

"New" ခလုတ်ကို နှိပ်ပြီးရင် Repository အသစ်ဆောက်မယ့် စာမျက်နှာကို ရောက်သွားပါမယ်။ အဲ့ဒီမှာ အောက်ပါအတိုင်း လုပ်ဆောင်ပါ -

  1. Repository name နေရာမှာ run-buddy လို့ အမည်ပေးပါ။
  2. တခြား Options တွေကို ဘာမှမပြင်ဘဲ "Create repository" ခလုတ်ကို နှိပ်လိုက်ပါ။

စာမျက်နှာအသစ်မှာ Quick Setup ဆိုတဲ့ Banner လေး ပေါ်လာပါလိမ့်မယ်။ အဲ့ဒီမှာ SSH ဆိုတာကို ရွေးပေးပါ။ လင့်ခ် (Link) က git@github.com:username/run-buddy.git ဆိုတဲ့ ပုံစံမျိုး ဖြစ်နေပါလိမ့်မယ်။

IMPORTANT

အကယ်လို့ Prework တုန်းက SSH keys တွေ သေချာလုပ်ခဲ့ရင် ဒီ SSH option ကိုပဲ သုံးပါ။ မလုပ်ရသေးရင်တော့ ရှေ့မဆက်ခင် SSH အရင်ပြင်ဆင်ထားဖို့ အထူးအကြံပြုပါတယ်။ GitHub tutorials on using SSH

Local Project နှင့် GitHub ကို ချိတ်ဆက်ခြင်း

အခု GitHub ပေါ်မှာ Remote repository ဆောက်ပြီးပြီဆိုတော့ ကိုယ့်စက်ထဲက Project နဲ့ ချိတ်ဖို့ လိုပါပြီ။ GitHub ကရတဲ့ Link ကို Copy ကူးပြီး Terminal (သို့မဟုတ် Git Bash) မှာ အောက်ပါအတိုင်း ရိုက်ပါ -

    git remote add origin [GitHub ကရတဲ့ Link ကို ဒီမှာထည့်ပါ]
    git remote -v

ဒုတိယ Command ဖြစ်တဲ့ git remote -v ကတော့ ကိုယ့် Project မှာ Remote link ချိတ်မိသွားပြီလားဆိုတာကို ပြန်စစ်တာပါ။ ဒီနေရာမှာ ကျွန်တော်တို့ ချိတ်လိုက်တဲ့လင့်ခ်ကို origin လို့ အမည်ပေးထားပါတယ်။ ဒီနာမည်က အရမ်းအရေးကြီးပါတယ်။ နောက်ပိုင်း Code တွေ တင်တဲ့အခါ ဒီနာမည်ကိုပဲ သုံးရမှာမို့လို့ပါ။

Code များကို GitHub ပေါ်သို့ တင်ခြင်း (Push)

အားလုံးအဆင်သင့်ဖြစ်ပြီဆိုရင် ကိုယ့်စက်ထဲက Code တွေကို GitHub ပေါ် တကယ်တင်ဖို့ အောက်ပါအတိုင်း ရိုက်ထည့်လိုက်ပါ -

    git push origin main

SSH ကို သုံးထားတယ်ဆိုရင် Git က မင်းရဲ့ SSH passphrase ကို တောင်းပါလိမ့်မယ်။ ပထမဆုံးအကြိမ် Push လုပ်တာဆိုရင် သင့်ကွန်ပျူတာက GitHub website ကို ယုံကြည်ရပါသလားလို့ မေးပါလိမ့်မယ်။ အဲ့ဒီအခါ yes လို့ ရိုက်ထည့်ပေးလိုက်ပါ။

git push origin main ဆိုတာ "ငါ့စက်ထဲက main branch မှာရှိတဲ့ code တွေကို origin (GitHub) ဆီကို ပို့ပေးပါ" လို့ ခိုင်းလိုက်တာ ဖြစ်ပါတယ်။

yes လို့ ရိုက်ပေးလိုက်ပါ၊ ကျွန်တော်တို့ သူတို့ကို ယုံကြည်လို့ရပါတယ်။ အကယ်လို့ HTTPS လင့်ခ်ကို သုံးမယ်ဆိုရင်တော့ GitHub ပေါ် Code တင်တိုင်း (Push လုပ်တိုင်း) သင့်ရဲ့ username နဲ့ password ကို အမြဲတမ်း ရိုက်နေရမှာပါ။ ဒါကြောင့် SSH ကို သုံးတာက ပိုပြီး အဆင်ပြေပါတယ်။

တကယ်တော့ git push origin main က ဘာလုပ်ပေးတာလဲ? git push ဆိုတဲ့ Command က ကိုယ့်စက်ထဲမှာ သိမ်းထားတဲ့ Local Commits တွေကို Remote location (ဒီနေရာမှာတော့ GitHub) ဆီကို ပို့ပေးလိုက်တာပါ။ လက်ရှိမှာတော့ ကျွန်တော်တို့ဆီမှာ Main Branch တစ်ခုပဲ ရှိပါသေးတယ်။

LEGACY LORE

အရင်တုန်းကတော့ Codebase တစ်ခုရဲ့ အဓိကအပိုင်းကို master လို့ပဲ ခေါ်ခဲ့ကြပါတယ်။ ဒါပေမဲ့ အခုနောက်ပိုင်းမှာတော့ main ဆိုတဲ့ အသုံးအနှုန်းက ပိုပြီး ခေတ်စားလာပါတယ်။ GitHub ကလည်း default အနေနဲ့ main ကိုပဲ သုံးသလို ကျွန်တော်တို့ သင်တန်းမှာလည်း main ကိုပဲ သုံးသွားမှာပါ။ ဒါပေမဲ့ အပြင်လောကမှာ ဝါရင့် Coders တွေက "master branch" လို့ ပြောနေကြတာကိုလည်း ကြားရနိုင်တယ်ဆိုတာ သိထားပေးပါ။

အခု Browser ထဲက မင်းရဲ့ GitHub Repository ကို သွားပြီး Refresh လုပ်ကြည့်လိုက်ပါ။ မင်းတင်လိုက်တဲ့ ဖိုင် ၂ ဖိုင်နဲ့ Commits မှတ်တမ်းတွေကို တွေ့ရပါလိမ့်မယ်။ ဒါဟာ တခြား ကွန်ပျူတာတစ်လုံးကနေပဲဖြစ်ဖြစ်၊ တခြား Developers တွေနဲ့ပဲဖြစ်ဖြစ် ဒီ Project ကို ဆက်လုပ်လို့ရသွားပြီဆိုတဲ့ အဓိပ္ပာယ်ပါ။

အမြဲတမ်း သုံးရမည့် Git Routine

အခုကစပြီး မင်းပြင်လိုက်တဲ့ Code တွေ GitHub ပေါ် အမြဲရောက်နေစေဖို့ အောက်ပါ Commands ၃ ခုကို အလွတ်ကျက်ထားဖို့ လိုပါတယ် -

  1. git add -A (ပြင်ထားသမျှ အကုန်လုံးကို စင်ပေါ်တင်မယ်)
  2. git commit -m "လုပ်ခဲ့တဲ့အလုပ်အကြောင်း ရေးရန်" (မှတ်တိုင်အဖြစ် သိမ်းဆည်းမယ်)
  3. git push origin main (GitHub ပေါ် တင်လိုက်မယ်)

Website ကို Live လွှင့်ခြင်း (GitHub Pages)

GitHub ရဲ့ အလန်းဆုံးအချက်ကတော့ ကိုယ့်ရဲ့ HTML Project တွေကို Live Website အဖြစ် အခမဲ့ တင်ပေးတာပါပဲ။ Hosting အတွက် ပိုက်ဆံပေးစရာ မလိုတော့ဘူးပေါ့။

ကျွန်တော်တို့ရဲ့ Run Buddy တိုးတက်မှုတွေကို တကယ့် Website လင့်ခ်အနေနဲ့ ကြည့်လို့ရအောင် အောက်ပါအတိုင်း လုပ်ဆောင်ပါ -

  1. ကိုယ့်ရဲ့ Repository စာမျက်နှာမှာရှိတဲ့ Settings tab ကို နှိပ်ပါ။
  2. စာမျက်နှာရဲ့ ဘယ်ဘက်ခြမ်းမှာရှိတဲ့ Pages ဆိုတဲ့ tab ကို ရှာပြီး နှိပ်ပါ။

GitHub Pages နဲ့ Live လွှင့်ခြင်း

GitHub Pages မှာ Source ကို "main branch" လို့ ရွေးပေးလိုက်တာဟာ GitHub ကို "ငါ့ရဲ့ main branch ထဲက code တွေကို သုံးပြီး website လွှင့်ပေးပါ" လို့ ပြောလိုက်တာ ဖြစ်ပါတယ်။ ဒါဆိုရင် GitHub က မင်းရဲ့ website လိပ်စာကို https://<your-username>.github.io/run-buddy/ ဆိုပြီး ပြပေးပါလိမ့်မယ်။

ဒါက တကယ် live ဖြစ်ဖို့ တစ်မိနစ်လောက် ကြာတတ်တာမို့လို့ ခဏစောင့်ပြီးမှ အဲ့ဒီ link ကို ဖွင့်ကြည့်ပါ။ ဖွင့်ကြည့်လိုက်တဲ့အခါ မင်းရဲ့ Run Buddy webpage က အင်တာနက်ပေါ်မှာ တကယ် ရှိနေတာကို တွေ့ရပါလိမ့်မယ်။

နောက်ပိုင်းမှာ ဒီ project ကို ပြင်ဆင်ပြီး GitHub ပေါ် push လုပ်လိုက်တိုင်း ဒီ live link က အလိုအလျောက် update ဖြစ်သွားမှာပါ။ တကယ့်ကို အဆင်ပြေတဲ့ ဝန်ဆောင်မှုပါပဲ။

ON THE JOB

Recruiters (အလုပ်ရှာပေးသူများ) နဲ့ Hiring Managers (အလုပ်ခန့်သူများ) အတွက် GitHub ဟာ အရမ်းအရေးကြီးပါတယ်။ မင်းရဲ့ coding လက်ရာတွေ၊ အလုပ်လုပ်ပုံတွေနဲ့ README ဖိုင်တွေကို ကြည့်ပြီး မင်းရဲ့ အရည်အချင်းကို အကဲဖြတ်ကြတာပါ။ ဒါကြောင့် GitHub profile ကောင်းကောင်းရှိဖို့ အခုကတည်းက ကြိုးစားပါ။

index.html ရဲ့ အရေးပါပုံ

မင်းရဲ့ link ကို ကြည့်ရင် index.html လို့ အနောက်မှာ ထည့်ရေးစရာမလိုဘဲ Website ပွင့်လာတာကို သတိထားမိမှာပါ။ ဒါဟာ Server တွေက ဖိုင်နာမည် သီးသန့်မဖော်ပြထားရင် index.html ကို default (အခြေခံဖိုင်) အနေနဲ့ အလိုအလျောက် ရှာဖွေပေးလို့ ဖြစ်ပါတယ်။

ဥပမာ - something.com/contact.html လို့ ရိုက်ရင်တော့ contact.html ဖိုင်ကို ပြပေးမယ်။ ဒါပေမဲ့ something.com လို့ပဲ ရိုက်ရင်တော့ Server က index.html ကိုပဲ ရှာပြီး ပြပေးမှာပါ။ ဒါကြောင့် Project တိုင်းမှာ index.html ဖိုင် ပါဖို့က အရမ်းအရေးကြီးပါတယ်။

Reflection (ပြန်လည်သုံးသပ်ခြင်း)

ဒီသင်ခန်းစာမှာ ကျွန်တော်တို့ Run Buddy အတွက် အခြေခံဖိုင်တွေ၊ Folder တွေဆောက်တာ၊ Git workflow သုံးတာနဲ့ GitHub repo ချိတ်ဆက်တာတွေကို အဓိကထား လုပ်ဆောင်ခဲ့ပါတယ်။ အခုကြည့်ရင် သိပ်မသိသာသေးဘူး ထင်ရပေမဲ့ တကယ်တော့ မင်းတို့ အများကြီး လုပ်ဆောင်နိုင်ခဲ့ပါပြီ -

  • HTML structure တစ်ခု အသင့်ဖြစ်နေပြီဖြစ်လို့ နောက်သင်ခန်းစာတွေမှာ အချက်အလက်တွေနဲ့ အလှဆင်ဖို့ပဲ လိုတော့တယ်။
  • Git ကိုသုံးပြီး ကိုယ့်ရဲ့ code ပြောင်းလဲမှုတွေကို ကိုယ့်စက်ထဲမှာရော GitHub ပေါ်မှာပါ စနစ်တကျ သိမ်းဆည်းတတ်သွားပြီ။
  • GitHub Pages ကိုသုံးပြီး ကိုယ့်ရဲ့ Website ကို အင်တာနက်ပေါ်မှာ Live လွှင့်ထားနိုင်ပြီ။

ဒီသင်ခန်းစာမှာ ဘာတွေတတ်မြောက်ခဲ့လဲ?

ဒီသင်ခန်းစာကို လေ့လာပြီးနောက်မှာ မင်းတို့အနေနဲ့ အောက်ပါအချက်တွေကို လုပ်ဆောင်နိုင်သွားပါပြီ -

  • touch နဲ့ mkdir command တွေကို သုံးပြီး ဖိုင်တွေ၊ folders တွေ ဆောက်တတ်လာတယ်။
  • cd နဲ့ cd .. ကို သုံးပြီး folder တွေထဲ သွားလာတတ်လာတယ်။
  • <html>, <head>, နဲ့ <body> တို့နဲ့ HTML structure ကို ချတတ်လာတယ်။
  • <div>, <section>, <header>, နဲ့ <footer> တို့ကို သုံးပြီး content တွေကို အပိုင်းလိုက် ခွဲတတ်လာတယ်။
  • Git မှာ git add နဲ့ Stage လုပ်တတ်လာတယ်။
  • git commit -m "" နဲ့ Commit လုပ်ပြီး git push origin main နဲ့ GitHub ပေါ် Code တွေ တင်တတ်လာတယ်။
